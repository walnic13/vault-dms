# Vault DMS Frontend — Layer 2: delta-based revalidation (patch, don't re-list) — Pass-1 Frontend Verified Evidence Pack

Plan-only VEP. Layers 1 + the SWR/re-show signal are live: the DMS tree paints instantly and revalidates by **re-listing** each expanded folder (`dms_tree` per node). Layer 2 makes revalidation **incremental** — the deployed `dms_delta` endpoint (API Spec §2.7) returns only what changed in a drive since a cursor, so on revalidation the client makes **one `dms_delta` call per expanded site** and **patches the cached tree in place** (by item id under parent id) instead of re-fetching every folder. First-load of a folder still uses `dms_tree` (lazy); delta is the revalidation path. Cheaper, faster, and still a true live mirror (every revalidation hits SharePoint via delegated OBO). Scope: `src/lib/dmsClient.ts` + `src/DmsBrowser.tsx`. No visual change, no new endpoint (§2.7 is deployed + golden-verified). Reviewer: Codex (FE).

## F-P2.5 — Gap Disclosure
**PROCEED.** (1) **Liveness preserved** — every revalidation still calls SharePoint via delegated OBO (`dms_delta`); nothing is served as authoritative-stale; per-user trimming is intact (the delta runs on-behalf-of the signed-in user, same as `dms_tree`). (2) **Root-parent reconciliation (the cache-key bridge).** The FE caches a folder's children under `treeKey(siteId, parentItemId)`, and the ROOT listing under `treeKey(siteId, undefined)` = `"${siteId}|"` (root `getDmsTree` is called with `parentItemId === undefined`). But `dms_delta` reports each change's `parent_id` from Graph `parentReference.id`, and for items DIRECTLY under the drive root that parent is the Graph **root item id**, not the empty root key. So the patch MUST translate: `applyDeltaToCache` maps a change whose `parent_id === rootItemId(siteId)` to the root cache key (`parentItemId` `undefined`); all other `parent_id`s map to `treeKey(siteId, parent_id)` directly. `rootItemId(siteId)` has EXACTLY ONE contract-grounded source: `dms_tree`'s root response `dms_tree.parent.item_id` (deployed API Spec §2.2 — always present on a root call). `getDmsTree` with no `parentItemId` captures it + persists it (per-site, namespaced snapshot). **Contract-safe root-unknown path (no reliance on an ungrounded delta-root item):** if `rootItemId(siteId)` is `null` when `revalidateSiteViaDelta` runs, it FIRST `await getDmsTree(siteId, undefined, getAccessToken)` — a root refresh that both sets `rootItemId` (via §2.2 `parent.item_id`) and refreshes the root listing directly — THEN applies the delta with `rootItemId` known. §2.7 does NOT guarantee a root item in `changes[]` (and the deployed handler skips a facet-less root item), so the plan relies solely on the §2.2 `parent.item_id` guarantee, never on a baseline-root delta change. (3) **Lazy/drive-level bridge for non-root folders** — the patch applies changes only to parent folders already present in `treeCache`; changes under not-yet-loaded folders are ignored (those load fresh via `dms_tree` on first expand). (4) **Resync-safe** — any `dms_delta` non-2xx (incl. expired-token 410→500) drops the stored cursor so the next revalidation re-baselines; a baseline is applied as upserts to loaded folders. (5) **Cursor + rootItemId storage** — both live in the principal-namespaced sessionStorage snapshot (DMS Snapshot Storage Exception, Governor §6.3 — "metadata + delta cursor" in scope). (6) **No new VA / no endpoint change** — §2.7 is deployed + golden-verified; `dms_tree.parent.item_id` is already in the deployed §2.2 response. No PRE-LAND / ESCALATE.
